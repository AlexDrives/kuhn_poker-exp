# Kuhn Poker 实验修改建议

这份建议主要面向当前仓库中的手写实验部分，目标不是简单“把代码调通”，而是把实验问题定义、实现正确性和评估指标整理清楚，让后续结论更可信、更可复现。

## 一、当前实验的主要问题

### 1. 实验目标和训练目标不一致

当前实验希望回答的问题是：

- 神经网络策略梯度方法能否在 Kuhn Poker 中逼近 Nash equilibrium

但当前训练方式实际上更接近：

- 对固定对手进行单边策略优化

这两者不是同一个问题。  
如果对手是 `RandomPolicy`，学到的更可能是 exploit random 的策略；  
如果对手是手写 `NashPolicy`，学到的也更接近该对手的 best response，而不是均衡求解。

### 2. 玩家位置语义和状态编码存在错位

当前环境中，每局会随机决定谁先手、谁后手，但状态编码里仍然使用固定的 `player_id` 作为位置特征。这样会导致：

- 同一个输入特征在不同对局中可能对应不同的真实博弈位置
- 训练样本的语义不稳定
- 策略网络学到的不是稳定的信息集映射

这会直接影响训练是否收敛，也会污染实验结论。

### 3. 评估方式不适合混合策略博弈

Kuhn Poker 的 Nash equilibrium 本质上是混合策略。  
当前 `main.py` 中使用 `greedy=True` 做评估，会把概率策略压成确定性策略。这样会带来两个问题：

- 即使模型已经学到接近均衡的混合分布，`argmax` 也会把它破坏掉
- 对战结果会低估模型真实策略质量

对于这类博弈，更合适的评估方式应该是：

- 按策略分布采样行动
- 计算 exploitability
- 按信息集比较动作概率

### 4. 手写 Nash 基线需要重新核对

当前 `NashPolicy` 的若干分支与 README 中描述的理论策略存在不一致的风险。  
如果基线本身不准确，那么：

- 训练目标会偏掉
- 评估对象也会偏掉
- 最终“和 Nash 的差距”这一结论会失去可信度

### 5. REINFORCE 实现过于原始

当前训练实现属于最基础的 Monte Carlo policy gradient：

- 整局统一 reward
- 没有 baseline
- 没有 advantage
- 没有 entropy regularization
- 没有多随机种子统计

这会导致梯度方差偏大，训练结果很容易受到随机性影响。

### 6. 缺少最小可信基线

Kuhn Poker 的状态空间非常小，本来非常适合先建立：

- 表格型策略基线
- 精确 exploitability 评估
- 自博弈训练对照

如果这些最小基线都没有建立，就很难判断问题到底出在：

- 算法目标
- 实现细节
- 函数逼近
- 评估口径

## 二、建议的修改优先级

建议按下面顺序推进，而不是直接继续堆神经网络训练轮数。

### 第一优先级：修正实验定义和环境语义

建议先做以下修改：

1. 固定玩家 0 为先手、玩家 1 为后手  
   或者显式把“当前是否先手”“当前 acting player 的真实位置”编码进状态。

2. 训练收益必须和训练主体严格对应  
   不要默认 `score[0]` 永远代表训练 agent 的收益，除非训练 agent 的先后手身份在整个实验中固定。

3. 重新核对并修正 `NashPolicy`  
   让理论基线和 README 中的说明保持一致。

### 第二优先级：重写评估口径

建议增加一个独立评估模块，至少提供以下能力：

1. 采样式对战评估  
   模型按概率采样动作，不使用 `argmax`。

2. 信息集策略打印  
   对 Kuhn Poker 的全部合法信息集打印 `CHECK / BET` 概率。

3. exploitability 评估  
   对于 Kuhn Poker 这种小游戏，可以手写枚举整棵博弈树，计算 best response 和 exploitability。

这样一来，手写部分就能和 OpenSpiel 的 CFR 部分使用同一套评价口径。

### 第三优先级：建立表格型基线

在引入更复杂的深度 RL 之前，建议先增加两个简单基线：

1. Tabular policy gradient
2. Tabular CFR / CFR+

理由：

- Kuhn Poker 的信息集数量极小
- 表格方法更容易调试和解释
- 可以先验证“算法机制对不对”，再验证“神经网络有没有必要”

## 三、推荐的下一步实验方案

### 方案 A：先做“最小可信版本”

目标：建立一个可信、可复现的手写 Kuhn Poker 实验框架。

建议步骤：

1. 修正环境中的先后手语义
2. 修正 `NashPolicy`
3. 改写 `main.py`，评估时使用采样策略
4. 新增 `exploitability` 计算脚本
5. 用固定随机种子跑多次实验并记录均值、标准差

完成后，你就能回答：

- 当前神经网络策略到底离均衡有多远
- 负收益是因为算法问题，还是因为实现问题

### 方案 B：加入 tabular 基线

目标：把“函数逼近问题”和“博弈学习问题”分离开。

建议新增：

- `tabular_reinforce.py`
- `tabular_eval.py`
- `tabular_cfr.py`

其中：

- `tabular_reinforce.py` 为每个信息集直接维护两个动作参数
- `tabular_cfr.py` 作为手写均衡求解基线
- `tabular_eval.py` 统一输出策略分布、对战收益和 exploitability

如果 tabular 版都不能稳定逼近合理策略，那么说明核心问题不在神经网络，而在训练目标或实验设计。

### 方案 C：改成自博弈实验

如果目标确实是研究“RL 能否逼近 Nash”，建议把训练从固定对手改为自博弈。

可以按难度分三步做：

1. 双边独立 REINFORCE
2. Self-play + opponent pool
3. Fictitious self-play 或 NFSP 风格近似

这样实验问题才真正从“打败某个对手”转向“逼近均衡”。

## 四、如何深化这个实验

### 1. 从单次结果改成系统对照实验

建议对比下面三组方法：

- Tabular REINFORCE
- Neural REINFORCE
- CFR / CFR+

统一比较：

- exploitability
- 与 Nash 的对战收益
- 信息集上的策略分布
- 多随机种子下的稳定性

### 2. 做消融实验

建议逐项分析以下因素的影响：

- 固定对手训练 vs 自博弈训练
- greedy 评估 vs sampling 评估
- 无 baseline vs 有 baseline
- 错误位置编码 vs 正确位置编码
- Random opponent vs Nash opponent

这样可以更准确地定位“到底是什么导致训练失败”。

### 3. 做轨迹级分析，而不仅仅看最终输赢

对于 Kuhn Poker，推荐记录：

- 每个信息集上的动作概率随训练轮数的变化
- exploitability 曲线
- 与不同基线对手的收益曲线

这会让实验结论更有解释力。

### 4. 扩展到更大的不完全信息博弈

如果 Kuhn Poker 阶段跑通，可以继续尝试：

- Leduc Poker
- 更大牌面或更多下注轮次的 toy poker game

这样就可以自然引出：

- 表格法何时开始失效
- 神经网络函数逼近何时真正有必要
- CFR 和 RL 方法在规模扩展下各自的优缺点

## 五、建议的短期路线图

如果只做一个短周期版本，建议按下面顺序：

1. 修正环境位置语义
2. 修正 `NashPolicy`
3. 去掉 greedy 评估
4. 新增 exploitability 评估
5. 增加 tabular baseline
6. 再重新训练和对比 neural policy

## 六、建议的中期研究问题

当基础框架稳定后，可以把实验问题升级为下面几类：

- 为什么单边 policy gradient 不擅长求零和不完全信息博弈均衡
- 在 Kuhn Poker 这种小游戏中，self-play 是否足以逼近 Nash
- 表格方法与函数逼近方法在小规模博弈中的误差来源分别是什么
- CFR 与 policy gradient 的差异究竟来自优化目标、策略表示，还是训练动态

## 七、总结

当前仓库已经有一个不错的出发点：

- 有手写环境
- 有神经网络策略训练
- 有 OpenSpiel 对照

但要让结论更扎实，下一步不应该继续直接“训练更多轮”，而应该优先完成三件事：

1. 修正实验定义
2. 建立统一且可信的评估口径
3. 增加表格型基线

等这三步完成后，再去讨论 REINFORCE、Actor-Critic、PPO 在这类博弈中的适用性，结论就会清楚很多。
